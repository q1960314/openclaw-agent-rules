# 执行保障体系 v2.0

## 📋 完整体系（2026-03-12 14:31）

### 1. 错误处理机制

| 工具 | 用途 | 位置 |
|------|------|------|
| **validate-report.py** | 验证汇报格式（5 要素） | `./validate-report.py` |
| **dispatch-task.py** | 分派任务（自动添加要求） | `./dispatch-task.py` |
| **midtask-check.py** | 中途检查（5 分钟后） | `./midtask-check.py` |

**流程：**
```
分派任务（明确要求）
  ↓
中途检查（5 分钟）
  ↓
收到汇报
  ↓
格式验证（5 要素）
  ↓
  ├─ 通过 → 接收
  └─ 不通过 → 打回重做
```

---

### 2. 优化建议处理

| 工具 | 用途 | 位置 |
|------|------|------|
| **optimization-handler.py** | 收集、分类、跟踪优化建议 | `./optimization-handler.py` |

**流程：**
```
子智能体发现优化
  ↓
按格式汇报（类别 + 优先级 + 描述 + 收益 + 建议）
  ↓
Master 记录（optimizations.json）
  ↓
分类处理：
  ├─ 高优先级 → 立即实施
  ├─ 中优先级 → 安排实施
  └─ 低优先级 → 记录待办
  ↓
实施后反馈
```

## 🎯 核心思维

### 错误处理
```
看到错误
  ↓
分析原因（为什么错？）
  ↓
尝试修复（怎么改？）
  ↓
验证结果（改好了吗？）
  ↓
汇报完整过程（问题 + 原因 + 修复 + 结果）
```

### 优化建议
```
发现优化点
  ↓
分析价值（有什么好处？）
  ↓
提出方案（怎么优化？）
  ↓
预估收益（提升多少？）
  ↓
汇报完整建议（类别 + 优先级 + 描述 + 收益 + 建议）
```

## 🔧 使用方式

### 分派任务
```bash
./dispatch-task.py test-expert "阶段 0.2 - 15 个接口测试"
```

### 验证汇报
```bash
./validate-report.py "汇报内容..."
```

### 添加优化建议
```bash
./optimization-handler.py add test-expert "性能优化" "可以并行执行" high
```

### 生成报告
```bash
./optimization-handler.py report
```
