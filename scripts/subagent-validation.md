# 子智能体任务验证机制 v2.0

## ❌ 之前的问题

| 问题 | 表现 | 原因 |
|------|------|------|
| **文档写了不等于执行** | AGENTS.md 有要求，子智能体不遵守 | 没有强制验证 |
| **汇报格式不检查** | 子智能体随便汇报，master 照单全收 | 没有验收标准 |
| **错误行为不纠正** | 看到错误不修复，也没有后果 | 没有惩罚机制 |
| **任务要求不具体** | "测试接口"太模糊 | 没有明确步骤 |

## ✅ 新的验证机制（4 层保障）

### 第 1 层：任务分派时的明确要求

**❌ 错误分派（太模糊）：**
```javascript
sessions_spawn({
  agentId: "test-expert",
  label: "master-test",
  task: "测试 15 个接口"
})
```

**✅ 正确分派（明确 + 示例）：**
```javascript
sessions_spawn({
  agentId: "test-expert",
  label: "master-test",
  task: `你是测试专家，负责接口测试。

## 核心要求（必须遵守）

1. **发现错误必须分析原因**，禁止只汇报"失败了"
2. **必须主动尝试修复**（换日期、改参数、重试等）
3. **必须验证修复结果**
4. **汇报格式**（强制）：
   - 问题现象
   - 原因分析
   - 修复尝试
   - 验证结果

## 示例（必须参考）

看到"非交易日"错误时：
- ❌ 禁止：汇报"空数据，测试结束"
- ✅ 必须：分析"可能今天数据没更新" → 换前一天测试 → 获取数据 → 汇报

## 验收标准

- 每个接口必须有明确的测试结果（成功/失败）
- 失败的接口必须有原因分析和修复尝试
- 汇报必须包含完整 4 要素

开始执行。`
})
```

### 第 2 层：汇报接收时的格式验证

**Master 收到汇报后的验证流程：**

```javascript
// Master 验证伪代码
function validateReport(report) {
  const required = ['问题现象', '原因分析', '修复尝试', '验证结果'];
  
  for (const item of required) {
    if (!report.includes(item)) {
      return {
        valid: false,
        reason: `缺少${item}`,
        action: '打回重新汇报'
      };
    }
  }
  
  return { valid: true };
}

// 如果验证失败
if (!validation.valid) {
  // 打回重新汇报
  sessions_send({
    sessionKey: subagentSession,
    message: `汇报格式不符合要求：
    
    缺少：${validation.reason}
    
    请重新分析并汇报，必须包含：
    1. 问题现象
    2. 原因分析
    3. 修复尝试
    4. 验证结果`
  });
}
```

### 第 3 层：中途检查点

**不等待最终汇报，中途检查进展：**

```javascript
// 分派任务后 5 分钟检查
setTimeout(() => {
  checkSubagentProgress(subagentSession);
}, 5 * 60 * 1000);

// 检查内容
function checkProgress() {
  // 1. 是否已经开始执行
  // 2. 是否遇到错误
  // 3. 遇到错误是否在主动修复
  
  // 如果发现偏离，立即纠正
  if (deviating) {

  }
```

### 第 4 层：违规惩罚机制

| 违规行为 | 惩罚措施 |
|---------|---------|
| 只汇报问题不分析 | 打回重新分析，明确批评 |
| 不主动尝试修复 | 指出错误，要求重新执行 |
| 汇报格式不符合 | 拒绝接收，要求按格式重新汇报 |
| 多次违规 | 记录到子智能体评估，影响下次任务分派 |

---

## 📋 实际执行流程

```
分派任务（明确要求 + 示例）
  ↓
子智能体执行
  ↓
中途检查（5 分钟后）
  ↓
  ├─ 正常 → 继续
  └─ 偏离 → 立即纠正
  ↓
收到汇报
  ↓
格式验证（4 要素检查）
  ↓
  ├─ 通过 → 接收并汇总
  └─ 不通过 → 打回重新汇报
  ↓
任务完成
```
