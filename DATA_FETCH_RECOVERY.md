## 📋 问题背景

**问题：** 多进程冲突导致数据覆盖，需重新抓取

- 无进程锁机制
- 并发配置过于激进（35 线程，4000 次/分钟）
- 保存频率低，数据丢失风险高

## 🔧 核心改进

### 进程锁机制
```python
import fcntl
lock_file = open('/tmp/fetch_data.lock', 'w')
fcntl.flock(lock_file, fcntl.LOCK_EX | fcntl.LOCK_NB)
```
- 排他锁：确保只有一个进程运行
- 非阻塞：获取不到锁立即返回
- 自动清理：进程退出时释放锁

### 保守配置对比

| 参数 | 原配置 | 新配置 | 改进 |
|------|--------|--------|------|
| 并发线程数 | 35 | 10 | ⬇️ 降低 71% |
| 限流（次/分钟） | 4000 | 2000 | ⬇️ 降低 50% |
| 保存频率 | 每 100 只 | 每 20 只 | ⬆️ 提升 5 倍 |
| 验证频率 | 无 | 每 100 只 | ✅ 新增 |

### 数据抓取优先级

| 优先级 | 数据类型 | 接口类型 | 说明 |
|--------|---------|---------|------|
| **P0** | 日线行情（daily） | 批量接口 | 最高优先级 |
| **P0** | 日线指标（daily_basic） | 批量接口 | 最高优先级 |
| **P1** | 财务数据 | 单股抓取 | 高优先级 |
| **P2** | 资金流向 | 单股抓取 | 中优先级 |
| **P3** | 龙虎榜/涨跌停 | 按日期抓取 | 低优先级 |

## ⚠️ 注意事项

### 1. 进程锁管理
- 正常退出会自动释放锁
- 异常退出后手动清理：`rm /tmp/fetch_data.lock`
- 启动前检查锁状态：`cat /tmp/fetch_data.lock`

### 2. 资源监控
```bash
# 监控 CPU/内存
top -p $(pgrep -f fetch_data_conservative)

# 监控网络
iftop -P -n
```

### 3. 日志位置
- 主日志：`logs/fetch_conservative_YYYYMMDD_HHMMSS.log`
- 验证日志：`logs/validate_YYYYMMDD_HHMMSS.log`
- 监控历史：`dashboards/monitor_history_*.json`

### 4. 异常处理
- 抓取失败率 > 5%：检查网络和 API 限制
- 进程意外终止：检查日志，清理锁文件后重启
- 数据验证失败：查看验证报告，手动修复或重新抓取
