# 限流算法优化说明 - 问题 3

## 优化文件
- `modules/data_fetcher.py`

## 优化内容

### 1. 【优化】令牌桶限流算法优化（更平滑）

**原算法问题：**
- 使用时间窗口硬重置方式，在窗口边界会产生请求尖峰
- 不允许突发流量，限流不够灵活

**优化后算法：**
- 采用标准令牌桶算法，令牌按固定速率持续补充
- 秒级令牌桶：每秒补充 `max_requests_per_second` 个令牌
- 分钟级令牌桶：每分钟补充 `max_requests_per_minute` 个令牌
- 允许突发流量（桶内有足够令牌时可立即处理多个请求）
- 更平滑的限流体验，避免时间窗口边界的请求尖峰

**核心改进：**
```python
# 按时间流逝补充令牌（而非硬重置）
self.second_tokens = min(max_rps, self.second_tokens + elapsed_second * max_rps)
self.minute_tokens = min(max_rpm, self.minute_tokens + elapsed_minute * (max_rpm / 60.0))
```

### 2. 【优化】添加限流状态日志（DEBUG 级别）

**新增日志内容：**
- 秒级限流触发时：记录当前令牌数、等待时间、累计触发次数
- 分钟级限流触发时：记录当前令牌数、等待时间、累计触发次数
- 日志级别：DEBUG（不影响正常 INFO 级别日志输出）

**日志示例：**
```
⏱️ 秒级限流触发：令牌=0.35, 等待=0.065 秒 [累计触发:15]
⏱️ 分钟级限流触发：令牌=0.82, 等待=0.492 秒 [累计触发:3]
```

### 3. 【优化】添加限流触发计数器

**新增计数器：**
- `second_rate_limit_count`: 秒级限流触发次数
- `minute_rate_limit_count`: 分钟级限流触发次数
- `total_request_count`: 总请求次数

**定期统计输出：**
- 每 1000 次请求输出一次限流统计
- 包含总请求数、各级限流触发次数、限流比例

**统计示例：**
```
📊 限流统计：总请求=1000, 秒级限流触发=45, 分钟级限流触发=3, 限流比例=4.80%
```

- `max_workers`: 20

- `max_requests_per_minute`: 2500
- `max_requests_per_second`: 自动计算（max_requests_per_minute // 60）

## 架构保持不变

- 保持原有的 `DataFetcher` 类结构
- 保持原有的 `_rate_limit()` 方法签名
- 保持原有的 `request_with_retry()` 调用方式
- 保持原有的线程锁保护机制

## 优化效果

1. **更平滑的限流体验**：令牌按时间持续补充，避免硬重置导致的请求尖峰
2. **更好的可观测性**：DEBUG 级别日志可随时开启，监控限流状态
3. **更易于调优**：计数器提供数据支撑，可根据限流比例优化配置参数
4. **允许合理突发**：桶内有令牌时可立即处理，提升高并发场景性能

## 使用方法

1. **开启 DEBUG 日志**：在配置中设置 `LOG_LEVEL = "DEBUG"`
2. **查看限流统计**：每 1000 次请求自动输出统计信息
3. **调优建议**：如果限流比例过高（>10%），可适当增加 `max_requests_per_minute`
