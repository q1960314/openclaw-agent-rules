## 二、模型选择原则（基于AutoGen最佳实践）

### 2.1 AutoGen的经验

```
AutoGen使用AgentTool模式：
- 每个agent可以有不同的model_client
- 复杂任务用强模型（gpt-4.1）
- 简单任务用快模型

LiteLLM的经验：
- 统一网关，后端可切换
- 支持负载均衡、fallback
- 不同模型适合不同任务
```

### 2.2 选择原则

| 原则 | 说明 |
|------|------|
| **任务复杂度** | 复杂推理用强模型，简单任务用快模型 |
| **上下文需求** | 长上下文任务用大窗口模型 |
| **代码能力** | 代码相关用coder专用模型 |
| **记忆卡住问题** | 选择不会过度加载记忆的模型 |

## 四、记忆卡住问题的解决方案

### 4.1 问题原因

```
某些模型（特别是reasoning模型）在启动时会加载大量历史记忆
导致循环调用memory_manager.py，最终超时卡住
```

### 4.2 解决方案

| 方案 | 说明 | 推荐 |
|------|------|------|
| **A. 限制记忆加载量** | 启动时只加载最近N条记忆 | ⭐⭐⭐ |
| **B. 使用不卡住的模型** | qwen3.5-plus、kimi-k2.5表现较好 | ⭐⭐ |
| **C. 禁用自动加载** | 仅在需要时手动加载（当前方案） | ⭐ |

### 4.3 推荐配置

```json
{
  "memory_config": {
    "auto_load": false,
    "max_memories_on_startup": 10,
    "lazy_loading": true
  }
}
```

## 七、延伸思考

### 7.1 模型切换机制

```
场景：agent执行复杂任务时可能需要切换模型

实现：
- 简单任务 → 快模型（qwen3.5-plus）
- 复杂推理 → 强模型（qwen3-max）
- 代码任务 → coder模型（qwen3-coder-plus）
```

### 7.2 成本优化

```
当前成本为0（bailian免费额度）

如果未来有成本：
- 简单任务用便宜模型
- 复杂任务用贵模型
- 需要负载均衡机制
```

### 7.3 A/B测试

```
可以同时测试多个模型配置：
- 组A：使用qwen3-max
- 组B：使用glm-5

比较执行效率和结果质量
```

## 八、总结

**核心思路**：
- OpenCode是能力放大器
- 不同agent需要不同的模型能力
- 根据任务特点选择最适合的模型

**关键变化**：
- coder用代码专用模型
- 策略/参数用推理强模型
- 数据/监控用稳定模型

**需要用户确认**：
1. 是否同意更新各agent的模型配置？
2. 是否有其他模型偏好？
3. 是否立即开始实施？
