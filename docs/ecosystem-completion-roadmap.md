# 从骨架到完整生态：细节步骤路线图

Updated: 2026-03-28 04:16 Asia/Shanghai

## 1. 目的

这份路线图用于回答两个问题：
1. 当前系统已经走到哪一步
2. 从当前“强骨架/可验证原型”到“更完整生态”，还需要哪些细节步骤

---

## 2. 当前状态定位

当前系统已经完成：
- worker 接入层面：核心角色已接入
- 执行层面：已有 9 个真实执行型 worker 跑通
- loop 层面：已有单 worker、多轮 retry、跨 worker 固定链、跨 worker 自适应链
- 治理层面：已有 audit / heartbeat / stale / lease / dashboard / governance actions
- 决策层面：已有 `decision_engine.py` 第一至第三轮收口

因此当前状态不应再被定义为“只有骨架”，而应定义为：

> **强骨架 + 多轮可验证原型 + 持续加固中**

---

## 3. 从现在到“更完整生态”的 6 大阶段

## 阶段 A：Master 深化（需求分析增强）

### 目标
让 `master-quant` 不只是分类/路由，而是更细致分析需求。

### 需要补的能力
1. 目标拆解
   - 主目标 / 子目标 / 约束 / 风险 / 验收标准 / 停止标准
2. 隐含需求识别
   - 收益优先 / 回撤优先 / 稳健优先 / 频率约束 / 风险边界
3. 依赖图生成
   - 哪些 worker 并行
   - 哪些 worker 串行
   - 哪些节点是必须先验证再继续
4. clarification 机制
   - 信息不全时先提问，而不是硬派单

### 完成标志
- `master` 具备结构化需求分析产物
- 能输出更完整的任务 DAG / 风险标签 / 验收口径

---

## 阶段 B：关键 agent 深度增强

### 目标
让关键 worker 从“能跑”进化到“做得更细”。

### 优先级最高的 3 个
1. `strategy-expert`
2. `parameter-evolver`
3. `factor-miner`

### 需要补的内容
#### strategy-expert
- 策略问题拆解模板
- 多方案比较
- 风险识别
- 验证路径设计

#### parameter-evolver
- 参数敏感度判断
- 稳健参数与脆弱参数区分
- 调参顺序
- 边界设计

#### factor-miner
- 因子假设模板
- 因子经济逻辑解释
- 失效场景分析
- 因子验证建议

### 完成标志
- 三个关键 agent 都具备“更细致的专家级输出模板”
- 其输出不只是 summary，而是结构化分析报告

---

## 阶段 C：决策中枢继续收口

### 目标
让 `decision_engine.py` 真正成为统一决策中枢，而不是只接管部分判断。

### 需要补的内容
1. 更细粒度的路由策略
2. loop / governance / routing 共用状态枚举
3. `manual_review_required` 的明确触发条件
4. 更统一的 stop / retry / block / pass 语义
5. 更复杂条件下的 adaptive path 规则

### 完成标志
- 关键决策不再散落在各个脚本里
- 主要流程都通过统一决策模块控制

---

## 阶段 D：持续运行化

### 目标
把系统从“可执行原型”推进到“更持续运行的生态”。

### 需要补的内容
1. worker loop / scheduler
2. health governance cron / daemon 化
3. 长期 lease / heartbeat 策略
4. stale 恢复策略分级
5. 巡检任务定期化

### 当前状态说明
- 这块目前只有脚本与能力雏形
- 还不能宣称已经常驻运行

### 完成标志
- 存在明确的常驻/定时驱动机制
- 系统可在更少人工触发下持续巡检/恢复

---

## 阶段 E：评测与打分体系

### 目标
从“会跑”升级到“知道自己有没有变强”。

### 需要补的内容
1. 关键 agent 评分标准
2. loop 效果评估
3. stop criteria 与质量标准联动
4. 深度能力评测题集
5. 回归验证集

### 完成标志
- 能用统一标准判断 agent 是否更细致、更稳、更有价值
- 不再只靠主观感觉评估进展

---

## 阶段 F：正式收口机制

### 目标
把原型系统推进到更接近长期使用的完整生态。

### 需要补的内容
1. 正式版本发布点
2. 人工审批点
3. 回滚机制
4. 变更记录
5. 候选结果与正式结果区分

### 完成标志
- 候选优化结果不会直接误当正式结果
- 正式收口保留明确的人类确认机制

---

## 4. 真实优先级（建议顺序）

### 第一优先级
1. `master` 深化
2. `strategy-expert` 深度增强
3. `parameter-evolver` 深度增强
4. `factor-miner` 深度增强

### 第二优先级
5. `decision_engine` 继续收口
6. stop criteria / adaptive policy 细化

### 第三优先级
7. 持续运行化
8. 健康治理深化

### 第四优先级
9. 统一评测 / 打分
10. 正式收口机制

---

## 5. 判断“是不是更完整了”的标准

系统不应以“接了多少 worker”作为唯一标准。
应至少同时满足：

1. **能运行**
2. **能验收**
3. **能继续多轮推进**
4. **能因不达标而停下**
5. **能识别并恢复异常**
6. **关键 agent 做得更细**
7. **master 能更细致地分析需求**
8. **正式结果有清晰收口点**

---

## 6. 当前最现实的下一步

如果现在继续做，我建议优先进入：

### 下一阶段
- `master` 需求分析增强
- `strategy-expert` 深度增强
- `parameter-evolver` 深度增强
- `factor-miner` 深度增强

原因：
- worker 接入层面已经够了
- 现在最缺的是“做得更细”
- 这会直接决定系统从“能跑”走向“更像成熟专家生态”

---

## 7. 当前最准确的一句话总结

> 系统的“接入与骨架”阶段已经基本完成；接下来真正决定是否能成为“完整生态”的，不再是多接几个 worker，而是 **master 深化 + 关键 agent 深度增强 + 持续运行化 + 统一决策中枢 + 正式收口机制**。
