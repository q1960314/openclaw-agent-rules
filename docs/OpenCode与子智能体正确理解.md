# OpenCode 与子智能体的正确理解

## 一、核心认知错误

### 我之前的错误理解

```
错误理解：
┌─────────────────────────────────────────┐
│ 14个子智能体 = 14个独立执行者            │
│ coder 会自己写代码                       │
│ backtest-engine 会自己运行回测           │
└─────────────────────────────────────────┘
```

### 正确的理解

```
正确理解：
┌─────────────────────────────────────────┐
│ OpenCode = 执行工具（Coding Agent）      │
│ 子智能体 = 独立会话（Session/角色）       │
│ master-quant = 协调者                    │
└─────────────────────────────────────────┘
```

---

## 二、各组件的真实身份

### 1. OpenCode 是什么？

```bash
# OpenCode 是一个 Coding Agent 工具
opencode run "实现板块轮动策略"  # 执行代码任务

# 类似于：
codex exec "实现板块轮动策略"
claude "实现板块轮动策略"
pi "实现板块轮动策略"
```

**特点**：
- ✅ 独立的执行工具
- ✅ 可以读写代码、运行测试、执行任务
- ✅ 支持后台运行（不阻塞会话）
- ✅ PTY 模式（完整的终端交互）

### 2. 子智能体是什么？

```python
# 从 agents_list 结果看
{
  "agents": [
    {"id": "master-quant"}  # 只有 master-quant 是配置的 agent
  ]
}

# 从 sessions_list 结果看
sessions = [
    {"key": "agent:coder:main"},           # 这是一个会话
    {"key": "agent:strategy-expert:main"}, # 这是一个会话
    ...
]
```

**特点**：
- ❌ 不是独立的 agent
- ✅ 是独立的**会话/Session**
- ✅ 有自己的对话上下文
- ✅ 可以接收消息、返回结果
- ❓ 执行能力取决于是否有工具权限

### 3. master-quant 是什么？

```
master-quant = 唯一配置的 Agent
├── 有 exec 工具（可以执行代码）
├── 有 sessions_send 工具（可以发送消息给子会话）
├── 有 message 工具（可以通知用户）
└── 是协调者角色
```

## 三、真正的架构问题

### 问题1：子智能体没有执行能力

```
当前状态：
用户: 派发任务给 coder
master-quant: sessions_send(sessionKey="agent:coder:main", message="实现板块轮动策略")
coder: 收到消息，然后... 怎么执行？

问题：coder 只是一个会话，它没有 exec 工具！
```

### 问题2：OpenCode 和子智能体的关系不清

```
疑惑：
- coder 应该调用 OpenCode 执行任务吗？
- 还是 master-quant 直接调用 OpenCode？
- 子智能体的价值是什么？
```

### 问题3：职能重复

```
master-quant 有 exec → 可以执行任何任务
OpenCode 也可以执行任何任务
子智能体存在但可能没用上
```

## 四、正确的架构设计

### 方案1：三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                      用户 (90)                               │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                  master-quant (协调者)                       │
│                                                             │
│  职责：                                                     │
│  - 接收用户指令                                              │
│  - 判断任务类型                                              │
│  - 路由到合适的执行者                                        │
│  - 汇总结果汇报用户                                          │
│                                                             │
│  工具：                                                     │
│  - sessions_send（派发任务）                                 │
│  - message（通知用户）                                       │
│  - exec（仅用于简单查询，不执行专业任务）                      │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│               子智能体会话（专业判断层）                      │
│                                                             │
│  coder: 判断代码任务细节，指导 OpenCode 执行                 │
│  strategy-expert: 判断策略逻辑细节                           │
│  test-expert: 判断测试验证细节                               │
│  ...                                                        │
│                                                             │
│  价值：专业判断 + 指导执行                                   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│               OpenCode（执行层）                             │
│                                                             │
│  执行实际的代码开发、测试、回测等任务                         │
│                                                             │
│  调用方式：                                                 │
│  opencode run "任务描述"                                    │
│  codex exec "任务描述"                                      │
│  claude "任务描述"                                          │
└─────────────────────────────────────────────────────────────┘
```

### 方案2：直接调用 OpenCode（简化版）

```
用户
  ↓
master-quant（判断任务类型）
  ↓
OpenCode/Codex/Claude Code（直接执行）
  ↓
master-quant（汇总结果通知用户）
```

**优点**：
- 简单直接
- 不需要子智能体中转

**缺点**：
- 子智能体变成摆设
- 缺少专业判断层

## 五、推荐方案

### 推荐：混合模式

```python
# 简单任务：master-quant 直接调用 OpenCode
用户: "帮我实现一个简单的函数"
master-quant: exec(command="opencode run '实现一个简单的函数'", pty=True)

# 复杂任务：派发给子智能体判断，再调用 OpenCode
用户: "实现板块轮动策略"
master-quant: sessions_send(sessionKey="agent:strategy-expert:main", message="设计板块轮动策略的逻辑")
strategy-expert: 返回策略设计方案
master-quant: exec(command="opencode run '按照以下设计方案实现...'", pty=True)
```

## 六、核心结论

### OpenCode 的作用

| 作用 | 说明 |
|-----|------|
| **执行工具** | 实际执行代码开发、测试、回测等任务 |
| **独立进程** | 可以后台运行，不阻塞会话 |
| **Coding Agent** | 类似于 Codex、Claude Code |

### 子智能体的作用

| 作用 | 说明 |
|-----|------|
| **专业判断** | 对任务进行专业分析 |
| **任务分解** | 将复杂任务拆解 |
| **指导执行** | 指导 OpenCode 如何执行 |
| **结果审查** | 审查 OpenCode 的执行结果 |

### master-quant 的作用

| 作用 | 说明 |
|-----|------|
| **协调者** | 接收指令、路由任务、汇总结果 |
| **用户窗口** | 唯一与用户对话的入口 |
| **调度中枢** | 调度子智能体和 OpenCode |

---

## 七、职能边界（最终版）

| 任务类型 | 执行路径 |
|---------|---------|
| 简单查询 | master-quant → exec (cat/ls/grep) |
| 简单代码 | master-quant → OpenCode |
| 复杂策略 | master-quant → strategy-expert → OpenCode |
| 代码开发 | master-quant → coder → OpenCode |
| 回测验证 | master-quant → backtest-engine → OpenCode/exec |
| 数据抓取 | master-quant → data-collector → exec |
